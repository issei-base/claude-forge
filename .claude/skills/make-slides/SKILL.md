# make-slides

渡された素材（メモ / 記事 URL / 設計書 / 箇条書き）を、**Marp で発表用スライド**に組み、PDF / PPTX / HTML に書き出す skill。狙いは「文字を流し込んだだけの密なスライド」ではなく、**1スライド1メッセージ**で口頭発表に耐える構成。

[[doc-illustrate]] との違い: あちらは**1枚もの**の図解 HTML（スクロールして読む閲覧資料・発表しない）。こちらは**複数ページ**の発表スライド（投影・配布する・PDF/PPTX 書き出し）。素材を「読み物」にするか「めくる資料」にするかで選ぶ。

同梱物（必ず使う。毎回テーマや構成を作り直さない）:
- `assets/theme.css` — forge テーマ（16:9・日本語フォント・配色・表紙 `lead` / 章扉 `section` / 情報過多 `tight` の class）。`@auto-scaling true` 宣言済みで**長いコード行・数式は横方向に自動縮小**される（縦の溢れは行数で対処）。**CSS は触らない**。色を変えるなら theme.css でなく **deck の frontmatter `style:` で `:root` 変数を上書き**する（原本を汚さない）。レンダリング時に `--theme` で渡す。
- `assets/template.md` — スライドの骨格（frontmatter + 表紙 + アジェンダ + 章扉 + 本文 + 2カラム + コード + まとめ）。**これをコピーして中身を差し替える**のが基本。
- `references/marp-syntax.md` — テンプレに無い記法（背景画像・fit・数式・発表者ノート・出力コマンド・VS Code 設定）の lookup。**必要になった時だけ**読む。

## Preflight

1. **marp があるか確認**: `which marp`。無ければ `npm i -g @marp-team/marp-cli` を案内して止まる（PDF/PPTX は Chrome も要る。通常はインストール済み）。
2. **素材を取得**（引数 or 直前の会話）。素材が無ければユーザーに聞く。URL の場合は本文を取得する（下記ワークフロー1）。
3. **出力形式を決める**: 既定は **PDF**（配布・投影に最も汎用）。「PowerPoint で渡したい/編集したい」なら PPTX、「ブラウザで見せる」なら HTML。**ユーザー指定があればそれに従い、無指定なら PDF + 編集したい時のために一声かける**。
4. **保存先は `~/Downloads/`**（repo 外なので公開リスクが無く、すぐ開ける・渡しやすい）。`mkdir -p ~/Downloads`。**claude-forge は public repo なので、生成した deck.md / 出力ファイルを repo 内の追跡される場所に絶対に置かない**。

## ワークフロー

### 1. 素材を読む

- **URL のとき**: 要約モデルを挟まない生テキストが最良。`curl -sL -A "Mozilla/5.0" <URL>` で取り、自分で読む（Claude docs は末尾 `.md`、AWS は本文が main 領域）。WebFetch は最後の手段（小型モデルが数値・固有名を幻覚する）。
- **ファイル / メモ / 箇条書きのとき**: そのまま読む。設計書なら「発表で何を伝えるべきか」に絞る（全文をスライドに写さない）。
- 素材の**幹**を掴む: 一番伝えたい結論は何か、聞き手は誰か、要点は 3〜5 個か。枝葉はスライドに載せず口頭・補足へ委ねる。

### 2. 構成を立てる（スライドに割る）

紙芝居の流れを先に決める。基本型は `assets/template.md` の並び:
1. **表紙**（`lead`）— タイトル + 一文サマリ + 発表者
2. **アジェンダ** — 章立て 3〜5 個
3. **章扉**（`section`）— 各章の頭
4. **本文** — 1スライド1メッセージ
5. **まとめ** — 要点の再掲 + 次アクション

### 3. deck.md を書く

- **テンプレをコピー**: `cp assets/template.md ~/Downloads/<slug>-<生成日>.md`。frontmatter（`marp: true` / `theme: forge` / `paginate` / `size`）は**ファイル先頭のまま触らない**（前に何か書くと無効化され YAML が本文に出る）。
- 各スライドを埋める。**スライドの作法（守る）**:
  - **1スライド1メッセージ。** タイトルはその主張そのものにする（「アーキテクチャ」より「処理は3層に分離する」）。1枚で言いたいことを1文で言えないなら割る。
  - **箇条書きは1枚6行・1行は短く（おおむね6語/30字）まで（6×6 ルール）。** 溢れるなら分割するか `<!-- _class: tight -->`。文章をそのまま貼らない。本文は左寄せ（中央寄せは表紙だけ）。
  - **同じレイアウトを3枚以上連続させない。** 箇条書き→2カラム→表→図と変化をつけて単調を避ける。
  - **章の頭に `section` 章扉**を入れて区切る。表紙は `lead` + `<!-- _paginate: skip -->`（採番から除外）、章扉は `<!-- _paginate: hold -->`（番号は出すが採番を進めない）。`false` は使わない（非表示でも採番が進み本文が2始まりになる）。
  - 対比は2カラム（テンプレの `columns`）、コードは ```` ```lang ````、料金・上限など「項目×値」は表で。
  - 図が要るなら画像 or `references/marp-syntax.md` の背景画像記法。**凝った作図は無理に作らず**箇条書き/表で代替してよい。**Marp 標準は Mermaid 非対応**なので Mermaid 記法は埋め込まない。
- **deck だけ色を変えたいときは frontmatter の `style:` で `:root` 変数を上書き**する（theme.css は触らない。記法は `references/marp-syntax.md`）。
- **数値・固有名・コマンド構文は素材どおりに。** 発表資料でも事実は曲げない。素材に無いことを盛らない。

### 4. レンダリングする

保存先で deck.md からビルド（`--theme` で必ず forge を渡す）:

```bash
marp ~/Downloads/<slug>-<日付>.md --theme .claude/skills/make-slides/assets/theme.css --allow-local-files --pdf -o ~/Downloads/<slug>-<日付>.pdf
```

- **`--allow-local-files` は必須に近い。** これが無いとブラウザ変換の制約で `./img.png` などローカル画像が PDF/PPTX/PNG に埋め込まれず**サイレントに欠落**する。公式は "NOT SECURE" 警告つきなので、**自分が用意した信頼できる deck にのみ**使う（出力先は repo 外の `~/Downloads/`）。
- `--pdf` を `--pptx` / 省略（HTML）に変えれば形式が変わる。複数欲しいと言われたら複数回叩く。PPTX を「編集できる形」で求められたら `--pptx-editable`（実験的・要 LibreOffice・再現性は通常 PPTX より低い）を条件付きで。
- theme.css のパスは**この skill の assets を指す**（symlink 経由でも実体に解決される）。

### 5. vision QA ループ（崩れが無くなるまで回す・省かない）

スライドは見た目が成果物。markdown を読むだけでは溢れ・はみ出し・コントラストは分からないので、**レンダリングした画像を実際に開いて検査し、直して再ビルドする**ループを回す:

1. 全ページを PNG 化: `marp <deck.md> --theme <theme.css> --allow-local-files --images png -o /tmp/qa.png`（`/tmp/qa.001.png` …）。
2. **Read tool で PNG を実際に開いて目視する**（最低でも表紙・各章扉・コード/表/2カラム・情報の詰まったスライド。枚数が少なければ全部）。次を確認:
   - テキスト・コード・図が**枠からはみ出していないか**、1枚に詰め込みすぎていないか。
   - 表紙 `lead` / 章扉 `section` の class が効いているか、ページ番号が表紙除外で始まっているか、日本語が化けていないか。
   - 文字が小さすぎ・薄すぎてコントラスト不足になっていないか。
3. **崩れていたら直して再ビルドし、もう一度 PNG を開いて確認する。** 直し方: 行数を減らす / `<!-- _class: tight -->` を当てる / スライドを分割する / 文を短くする。**崩れが無くなるまで 2〜3 を繰り返す**（一度レンダリングして終わりにしない）。
- 仕上がったら出力形式（PDF 等）を本ビルドし、**自動で開く**: macOS は `open <出力ファイル>`。
- ユーザーに **保存先パス・スライド枚数・構成（章立てと各章の要点）・出力形式**を報告し、「枚数や粒度・別形式（PPTX 等）に直す?」と一声かける。元の deck.md（`.md`）も残してあるので**そこを直して再ビルドすれば微修正できる**ことを添える。

## してはいけないこと

- **frontmatter をファイル先頭から動かさない。** 前に空行・コメント・本文を置くと無効化され、`marp: true ...` が本文として表示される（既知の落とし穴）。
- **theme.css の CSS を手で打ち直さない。** deck の色を変えるなら frontmatter の `style:` で `:root` 変数を上書きする（原本 theme.css は触らない）。インラインに hex を直書きするとデザインがドリフトする。
- **1スライドに詰め込まない。** 文章ベタ貼り・10行超の箇条書きは分割する。読ませる資料が欲しいなら [[doc-illustrate]] へ。
- **素材に無い数値・固有名・主張を盛らない。** 発表用でも事実は曲げない。URL は生テキストで裏取りし、WebFetch の要約を鵜呑みにしない。
- **生成物を git に commit しない。** 出力先 `~/Downloads/` は repo 外（public repo に追跡させると公開される）。repo 内に出すなら必ず `.gitignore` 済みの場所に。
- 受講生の個人情報・非公開 URL を素材から持ち込んで永続保存しない。
