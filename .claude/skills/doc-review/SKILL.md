# Technical Document Review Workflow

技術ドキュメント・実装計画ドラフトを、**複数の専門家レビュワーを並列にかけて**多観点レビューし、リーダー（スキル実行者）が指摘を集約・修正して再レビューする。レビュワーが納得するまで（最大3ラウンド）サイクルを回す。

> **このスキルは Agent tool ベースのオーケストレータ。** 中核のドキュメント品質レビューは同梱の [`doc-reviewer`](../../agents/doc-reviewer.md) agent に委譲し、それだけでは拾えない専門観点（security / ops / cost）が必要なときだけ `general-purpose` レビュワーを追加で並列起動する。リーダーが全レビュワーの返り値を集約し、修正を実施し、再レビューを回す。
>
> **doc-reviewer agent との使い分け:** ドキュメントを 1 回サッと見てほしいだけなら `doc-reviewer` agent を直接呼ぶ（軽い）。**複数観点 + 修正ループ**まで回したいとき、または plan-issue から計画ドラフトのレビューを委譲されたときに、このスキルを使う（重い）。

## Triggers

- 重要な技術ドキュメントを多観点でレビュー＋修正ループしたいとき
- plan-issue からの計画ドラフトレビュー委譲（ファイルパス or テキストで対象が渡される）
- `/doc-review`

## レビュー対象の受け取り方

対象は以下のいずれかで渡される。最初に対象を確定する。
- **ファイルパス**（例: `docs/foo.md`、`/tmp/plan_123.md`）→ レビュワーに Read させる
- **インラインテキスト**（plan-issue がドラフト本文を直接渡す等）→ レビュワーの prompt に本文を埋め込む

## Process Flow

```
1. レビュー対象の確定（ファイルパス or テキスト）
2. 観点の選定: 中核 = doc-reviewer agent。追加 = security/ops/cost（内容に応じて）
3. Agent tool で「同一メッセージ内で並列起動」
     - subagent_type: doc-reviewer （中核のドキュメント品質レビュー）
     - subagent_type: general-purpose （security/ops/cost が必要な場合のみ）
     - subagent_type: fact-checker （数値・上限・価格・構文など検証すべき事実主張が多い場合のみ）
       ↓ 各レビュワーが構造化レビュー結果を返す
4. リーダーが結果を集約・妥当性判断
       ↓
   ┌─────────────────────────────────────┐
   │ 全員 PASS → 完了レポート              │
   │ WARN/FAIL あり → 修正して再レビュー    │
   └─────────────────────────────────────┘
5. リーダーがドキュメント/ドラフトを修正（レビュワーは修正しない）
6. 再起動する観点を Step 3 の手順で選定して再起動（指摘のあった観点＋修正が波及する観点。前回指摘と修正内容を prompt に渡す）
7. PASS になるまで繰り返し（最大3ラウンド）
8. 完了レポート出力
```

## 役割分担

### Leader（自分自身 = スキル実行者）

- レビュワーの選定・並列起動
- レビュー結果の妥当性判断（的外れな指摘は理由付きで却下）
- ドキュメント/ドラフトの修正実施
- 再レビューの起動
- 完了レポートの作成

### Reviewers

| 役割 | 起動方法 | カバー観点 |
|------|---------|-----------|
| **中核レビュワー** | `subagent_type: doc-reviewer` | 技術的正確性・完全性・明確性・実用性・構成（一次ソース裏取り込み） |
| **security 専門** | `subagent_type: general-purpose`（下記 prompt） | 脅威モデル・認証認可・機密情報・リスク明記漏れ |
| **ops 専門** | `subagent_type: general-purpose`（下記 prompt） | 運用手順の実行可能性・障害復旧・監視・ロールバック |
| **cost 専門** | `subagent_type: general-purpose`（下記 prompt） | 実装工数・代替比較・ランニングコスト・ROI |
| **fact 専門** | `subagent_type: fact-checker` | 数値・上限・クォータ・価格・コマンド/API 構文など事実主張を claim 単位で一次ソースに照合（VERIFIED/REFUTED/UNVERIFIABLE・出典付き） |

**レビュワーはドキュメントを直接編集しない** — 指摘と改善案を返すだけ。修正はリーダーが行う。
**内容に応じて必要なレビュワーだけ起動する**（下記 Reviewer Selection Guide）。個人開発のドキュメントでは過剰に全観点を起動しない。

## Step 1: レビュワー起動（並列）

選定したレビュワーを **同一メッセージ内で Agent tool を複数回呼び出して並列起動**する。

```
Agent tool（同一メッセージで並列）:
  # 中核（ほぼ常に起動）
  subagent_type: doc-reviewer
  prompt: |
    以下のドキュメントをレビューしてください。
    対象: <ファイルパス or 本文>
    （計画ドラフトの場合）これは plan-issue が Issue コメントとして投稿予定の実装計画です。
    前回の指摘（あれば）: <...>

  # 専門観点（必要な場合のみ追加）
  subagent_type: general-purpose
  prompt: <Specialist Prompt Template に観点を埋めたもの>
```

各レビュワーは**最終メッセージとしてレビュー結果を返す**。リーダーはその返り値を受け取って集約する。

## Step 2: リーダーの結果集約・修正

1. **結果受信**: 各レビュワーの返り値を読む
2. **妥当性判断**: 指摘が的確かを判断
   - 的外れ・スコープ外の指摘は対応不要と判断し、理由を記録（完了レポートに残す）
   - 妥当な指摘は修正対象としてマーク
3. **修正**: critical → major → minor の順に対応
4. **計画ドラフトのレビュー時の注意**: plan-issue から委譲された計画ドラフトでは、ops 観点の「ロールバック手順を追加」「監視チェックリストを追加」「進捗管理表を追加」など**コメントを長大化させる提案は原則却下する**（plan-issue §4-1 の「原則含めないセクション」方針）。却下理由を完了レポートに残す。

## Step 3: 再レビューサイクル

- 再起動する観点は次の手順で選ぶ: ①指摘のあった観点は必ず再起動 → ②今回の修正内容を見て、**修正が他の観点の判定を変えうるならその観点も再起動** → ③どちらにも当たらない観点は再起動しない。②の典型 (再起動単位はレビュワー = 中核/security/ops/cost/fact): 運用手順・構成を書き換えた → ops も / 認証・機密情報に触れる記述を変えた → security も / 実装方式を変えて工数・費用が動いた → cost も / 専門観点 (security/ops/cost) の指摘対応で本文を書き換えた → 中核 (doc-reviewer) も。中核は 1 起動で品質 5 軸 (正確性・完全性・明確性・実用性・構成) を全て見るので、軸単位の再起動は無い。「前ラウンドで PASS したか」ではなく「今回の修正を新規に検証すべき観点はどれか」で判断する。
- 再起動は Agent tool で新規起動し、prompt に「前回の指摘」と「今回の修正内容」を含めて渡す。
- 各観点が PASS を返すまで繰り返す。
- **最大3ラウンド**。3ラウンド目でも FAIL の観点があれば、残課題を完了レポートに明記して終了。

## Status Definitions

| ステータス | 意味 | 次のアクション |
|-----------|------|--------------|
| **PASS** | critical/major問題なし | 完了（その観点は終了） |
| **WARN** | major問題あり、criticalなし | 修正して再レビュー |
| **FAIL** | critical問題あり | 修正して再レビュー |

## Severity Definitions

| 重要度 | 定義 | 対応 |
|--------|------|------|
| **critical** | 技術的に誤った情報 | 必須修正 |
| **major** | 重要情報の欠落 | 必須修正 |
| **minor** | 品質改善可能 | 推奨修正 |
| **suggestion** | より良くする提案 | 任意修正 |

## Leader's Judgment Guidelines

### 採用する指摘
- 具体的な根拠（公式ドキュメントURL等）が示されている
- ドキュメントの目的・対象読者に合致している
- 再現可能な問題を指摘している

### 却下する指摘
- ドキュメントのスコープ外の指摘
- 過度に細かい・些末な指摘（suggestion でも対応コストが高い）
- 根拠が不明確または推測に基づく指摘
- （計画ドラフトの場合）コメントを長大化させる ops 提案

### 矛盾する指摘への対応
- 複数レビュワーの指摘が矛盾する場合、リーダーが最終判断し、理由を完了レポートに記録

## Reviewer Selection Guide

中核（doc-reviewer agent）は基本常に起動。専門観点は内容に応じて追加する。

| ドキュメント種別 | 中核 | 追加する専門観点 |
|----------------|------|----------------|
| インフラ設計書 | doc-reviewer | security, ops, cost |
| 運用ガイド | doc-reviewer | ops |
| セキュリティ設計 | doc-reviewer | security, ops |
| API設計書 | doc-reviewer | security |
| 技術選定レポート | doc-reviewer | cost |
| 実装計画ドラフト（plan-issue） | doc-reviewer | security/ops（計画の内容に応じて。無くてもよい） |
| 一般技術ドキュメント・教材 | doc-reviewer | （追加なし。事実主張が多ければ fact） |

> **fact をいつ足すか**: 数値・上限・クォータ・価格・コマンド/API 構文といった**検証すべき事実主張が多い**文書（教材・調査レポート・API 設計書など）は fact 専門（fact-checker）を追加する。中核の doc-reviewer も一次ソース裏取りをするが、**文書まるごとの品質＝doc-reviewer / claim 単位で全数を逐語照合＝fact-checker** と使い分ける。AWS の数値・挙動が主眼なら [[aws-docs]] で足りることも多い。

## Iteration Limit

- 最大レビューラウンド: **3回**
- 3回目でもFAILのレビュワーがいる場合: 残課題を明記して完了

### 専門観点パラメータ

#### security
- **PERSPECTIVE**: セキュリティ / **PERSONA**: CISSPホルダー・ペネトレーションテスト経験者
- **REVIEW_POINTS**: 脅威モデルの適切性 / 認証・認可の考慮 / 機密情報の取り扱い / セキュリティリスクの明記漏れ / 対策の有効性と検証可能性

#### ops
- **PERSPECTIVE**: 運用・可用性 / **PERSONA**: プラットフォームエンジニア（10年以上）
- **REVIEW_POINTS**: 運用手順の実行可能性 / 障害時の影響範囲と復旧手順 / モニタリング・アラート設計 / スケーラビリティ / ロールバック手順の有無

#### cost
- **PERSPECTIVE**: 実装コスト・費用対効果 / **PERSONA**: テックリード / エンジニアリングマネージャー
- **REVIEW_POINTS**: 実装工数の妥当性 / 代替手段との比較 / ランニングコスト（インフラ費用等）/ 技術的負債のリスク / ROI の明確性

## Notes

- レビュワーは結果のみを返し、ドキュメントの直接編集は行わない。修正の最終判断と実施はリーダーが担う。
- レビュワー間で矛盾する指摘はリーダーが裁定する。
- 公式ドキュメントで確認できない挙動は「要検証」としてマークする。
- 過度な品質追求よりも、実用的なドキュメントを優先する（個人開発スコープで価値に見合わない指摘は却下してよい）。
- ドキュメントの目的（調査メモ vs 公式ガイド）に応じてレビュー厳格度を調整する。
