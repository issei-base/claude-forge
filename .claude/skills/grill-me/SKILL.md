# grill-me（尋問モード）

生煮えの計画・アイデアを、共通理解に達するまで**一問一答で執拗にインタビュー**して固める。成果物は決定の集合（決定サマリー）であって、コードではない。**このスキルは実装しない。** コーディング以外（企画・PRD・記事構成）にも使う。

## 手順

1. **対象を把握し、事実を先に埋める。** 渡された計画/アイデアを読む。既存コードベースが関わるなら、質問を始める前に関連コード・既存パターンを読む。新規開発（コードが無い）や非コーディング（企画・PRD・記事構成）なら、代替の事実源（repo 内の関連ドキュメント・過去の類似企画・類似実装の WebSearch）を先に当たり、無ければこのステップは省略して次に進む（「調べようがない」と「調べ忘れ」を区別する）。**コードや資料を調べれば分かる「事実」は質問しない — 自分で調べて埋める。issei に問うのは「決定」（トレードオフ・優先度・スコープの選好）だけ。**
2. **決定木を組み立て、手戻りコストが高い分岐から聞く。** 決めるべき分岐を洗い出して依存順に並べ、**「後続の決定がいくつこの選択に依存するか」を基準に、間違えたときに戻るのが高くつくもの**（コーディングならデータモデル・スコープ境界・外部依存、企画なら対象範囲・対象レベル・実施形態）から質問する。UI の細部・命名などの低コスト分岐は後回し。issei の計画に書かれていない分岐（エラー時・権限・既存機能との衝突・後方互換）も能動的に立てる — 書かれたことをなぞるだけの尋問にしない。対話中に新しい分岐を見つけたら決定木に足し、次のチェックポイントで残り見込みを更新する（洗い出しは一度で確定させない）。
3. **質問は 1 問ずつ。** `AskUserQuestion` の `questions` 配列には**要素を必ず 1 件だけ**入れる（複数質問をタブでまとめて出せるが使わない）。1 回 1 問・2〜4 択、**推奨案を必ず先頭**に置き、推奨理由と対抗馬の差分を短く添える。回答が来るまで次の質問に進まない。**関連する論点が近くても依存順に分割し、1 ターンにまとめて投げない**（回答者を混乱させる）。なぜ今この質問か（何がこれに依存するか）を 1 行で示す。
4. **矛盾を検知したら即指摘する。** 新しい回答が過去の決定と衝突したら、黙って進めず「Q2 で A と決めたがこの回答は B を含意する。どちらを取るか」と確認してから先に進む。
5. **5 問ごとにチェックポイント。** 決定木マップ（✅決定済み / ⏳未決・残り何問の見込み）を短く提示し、「続ける / ここで畳む」を選ばせる。畳む場合も未決分岐を必ず明記する — 高コスト分岐から聞いているので、途中で畳んでも重要な決定は残っている状態にする。
6. **決定サマリーで締め、橋渡しする。** 全分岐の解決（または畳み）後に、①決定した事項と理由 ②未決のまま残した分岐 ③置いた前提、をまとめて提示する。その上で次を選ばせる: 決定を repo の ADR（`docs/adr/`）に書き出す / [[plan-issue]] で Issue の実装計画に落とす / [[banso]] や [[implement-issue]] で実装に進む / ここで終わる。対象が非コーディング（企画・PRD・記事構成）なら ADR・実装系 skill は提示せず、「決定サマリーを指定ファイルに保存する / ここで終わる」だけを選ばせる。**ADR 書き出し・保存は選ばれたときだけ**行う（勝手にファイルを作らない）。

## やらないこと

- **実装しない。** 共通理解に達しても、issei が明示的に後続へ進めない限りコードを書かない。
- **複数質問をまとめない。** 1 ターン 1 問が本体。効率のためにまとめたくなっても崩さない。
- **事実を質問しない。** 調べれば分かることを聞くのは尋問の質を下げる。調べてから聞く。
- **質問のための質問をしない。** どの決定にも繋がらない確認は積まない。全分岐が解決したら粘らず畳む。
