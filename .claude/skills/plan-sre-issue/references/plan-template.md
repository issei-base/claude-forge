# 実装計画書テンプレート

`plan-sre-issue` が Issue コメントとして投稿する実装計画の雛形。以下をベースにしつつ、Issue の内容に応じてセクションを追加・削除する。

---

```markdown
# 実装計画: <Issueタイトル>

- **Issue**: <Issue URL>
- **作成日**: <yyyy-mm-dd>
- **ステータス**: ドラフト

## 1. 背景・目的
Issueの要約と、なぜこの変更が必要かを簡潔に記述する。

## 2. スコープ
### やること
- 箇条書きで列挙
### やらないこと
- 明示的に除外するものを列挙（別 Issue で扱う事項）

## 3. 前提条件・制約
インタビューで確認した前提や技術的制約を記述する。

## 4. 影響範囲
### 変更対象ファイル
| ファイル | 変更内容 |
|---------|---------|
| `path/to/file` | 変更の概要 |
### 影響を受けるコンポーネント
- 依存する既存コード（呼び出し元・参照元）への影響を記述

## 5. 実装方針
### アーキテクチャ・設計
技術的なアプローチ、採用するパターン、データフローなどを記述する。
必要に応じて図やコード例を含める（コード例はコピペで使える粒度で）。
### 実装ステップ
1. **ステップ1**: 概要
   - 詳細な作業内容
   - 対象ファイル
2. **ステップ2**: 概要
   - 詳細な作業内容
   - 対象ファイル

## 6. テスト計画
### ユニットテスト
- テスト対象と観点
### 結合テスト / E2Eテスト
- テストシナリオ
```

---

## テンプレート使用時の注意

- セクションは Issue の規模や性質に応じて柔軟に調整する。
- 小さなバグ修正なら「実装方針」と「テスト計画」だけで十分な場合もある。
- コード例は必要最小限に留め、方針の説明を優先する。
- **SKILL.md §4-1 の「原則含めないセクション」を雛形に足さないこと** — ロールバック手順 / 検証チェックリスト / 進捗チェックリスト / 詳細なリスク・懸念表 / 監視ダッシュボード URL 等の運用詳細は、実装計画コメントには含めず、実作業時のプレイブックで扱う。上の雛形に「リスク・懸念」表を置いていないのはこのため。
- 目安は全体 150-200 行。第三者がこのコメントだけ読んで着手できる粒度を維持する。
