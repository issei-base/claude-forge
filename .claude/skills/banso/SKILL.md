# banso（伴走モード）

自律で進めさせるが、**要所で止まって issei を巻き込む**ことで「issei の盲点を炙り出し・知見が溜まる」ことを狙う作業モード。成果物そのものより **issei が理解して意思決定した状態で終わること**が目的。実装・設計・リファクタ・調査、どんなタスクにも被せて使う。

`implement-issue` が「止まらず完走」なら、banso はその**対**。要所のゲートで必ず止まり、issei の判断を仰いでから進む。

## いつ止まり、いつ進むか

**5 つのゲートで止まる。それ以外は自律で進める。** ゲートを飛ばさない — 飛ばすと「知見が溜まる」という目的自体が崩れる。

```
- [ ] Gate 1: 盲点マップ + 曖昧点の質問（着手前・止まる）
- [ ] Gate 2: 複数案をプロトタイプで提示（止まって選ばせる）
- [ ] Gate 3: 実装計画を確認（承認を取ってから実装）
-      （実装 — ここは自律で進める）
- [ ] Gate 5: 理解確認（最後・止まる）
-  ↳ Gate 4: 知見メモ（Gate 1〜5 を貫く継続スレッド・止まらない）
```

このチェックリストをレスポンスにコピーし、各 Gate 完了ごとにチェックを入れて進める。

## Gate 1: 盲点マップ + 曖昧点の質問（着手前）

**まず盲点マップを出す**（例外は 1 つ — 経験値を判定できない時だけ、下の較正の予備質問 1 問がマップに先行する）。タスクを実装に落とす前に、issei が見落としていそうな論点を炙り出して短く提示する。これが伴走の中核。3〜6 個に絞る（根拠を示せる盲点がそれに満たない小規模タスクでは、無理に埋めず出せる件数に留める）:

- 暗黙の前提（「たぶんこう想定してる」を明文化し、違ったら言ってもらう）
- 抜けがちな観点（エラー時・空データ・権限・既存機能との衝突・後方互換）
- 決めきれていない選択肢（後で手戻りになる分岐）
- スコープの境界（やる/やらない。過剰実装の芽を先に潰す）

**盲点は推測で列挙せず、関連コードを実際に読んでから出す**（読む対象がまだ無い新規タスクなら類似実装・規約を読む）。コードを読まない盲点マップは一般論の落とし穴リストに堕ちる。各項目に確信度（高/中/低）と根拠（`ファイル:行`）を 1 行で添え、根拠を出せない項目は「推測」と明記する。

**盲点マップは issei の当該領域の経験値に合わせて較正する。** 未知の領域なら unknown unknowns の解説（何を知らないか・何を聞くべきか）に寄せ、熟知した領域なら trade-off の指摘に寄せる。経験値は会話中の言及と memory を実際に確認して判定し、高確信度で判定できる場合（issei が当該領域を作り込んだ形跡がある・memory に明示の事実がある）のみ較正して進む。それ以外は判断できないとみなし、マップを出す前に独立した `AskUserQuestion` 呼び出しで当該領域への馴染みだけを 1 問先に聞く（下の「決めるべきことだけ」の質問予算とは別枠・別呼び出しの予備質問。馴染み以外をここで先に聞かない）。

> テキストの質問で出せる盲点はここまで（ただし「見本にしたい参考実装はあるか」のように既存コードを実物で指させる質問は、言葉にさせずに済むので例外 — 下の質問候補参照）。それ以外の「見れば分かるのに先に言葉にできない基準」（無自覚の既知）は Gate 2 のプロトタイプで初めて出てくる — Gate 2 を省くか迷ったらこの点も考慮する。

**次に、決めないと手戻りする論点だけを質問する。** 質問は `AskUserQuestion` で最大 4 問・各 2〜4 択にまとめ、必ず推奨案を先頭に置く。問いの並びは手戻りの大きい順 — 回答次第でアーキテクチャ・データモデルが変わるものを 1 問目に置く。「なんでも聞く」のではなく、**issei が決めるべきこと（好み・優先度・トレードオフの取り方）だけ**に絞る。コードを見れば分かること・合理的な既定があることは聞かず、自分で決めて Gate 3 の計画に明記する。言葉にしにくい要件（見た目・使い勝手・「あんな感じ」）が疑われるときは「見本にしたい既存実装・参考コードはあるか」を質問候補に含める — 参照コードは無自覚の既知を伝える最良の手段で、説明やスクリーンショットより情報量が多い。

## Gate 2: 複数案をプロトタイプで提示

**1 案で即実装しない。2〜3 案を「最小の動くプロトタイプ」で見せて選ばせる。** プロトタイプは完成品ではなく、違いが分かる最小形:

- コードなら**擬似コード or 核だけの数十行**（全実装しない）
- UI なら**HTML/SVG のモック**（`Artifact` か `mcp__visualize__show_widget` で見せる）
- データ設計・フローなら**図 or スキーマの骨子**

各案に **trade-off 表**（実装量 / 拡張性 / リスク / 既存との整合）を必ず添え、**推奨案とその理由**を明言する。issei が案を選ぶ（または混ぜる）まで実装に入らない。

**issei がその領域の良し悪しの基準を持っていないと分かったら（Gate 1 の較正で未知領域と判定した場合を含む・「何が良いか分からない」と言う・選ぶ基準を示せない）、issei の発言を待たず、案を並べる前にその領域の判断軸を短くレクチャーして基準を作ってから見せる。** 基準が無いまま案を並べても選べず、Gate 2 が当てずっぽうの選択に堕ちる。

> タスクが小さく案が実質 1 つなら、その旨を述べて Gate 2 を省く。ただし「他に筋がないこと」を 1 行で示してから進む（黙って飛ばさない）。

## Gate 3: 実装計画を確認

選ばれた案について、実装前に計画を出して**承認を取る**（implement-issue と違い、ここは止まる）:

1. **やること / やらないこと** — スコープ境界
2. **変更するファイルと順序** — 依存を考慮したステップ
3. **Gate 1 で決めた判断** — 何をどう決めたか
4. **検証方法** — どう動作確認するか

**提示する前に、第三者のシニアエンジニアとして自分で計画を批判的にレビューする。**「このまま進めると後で高くつくズレ」（データモデルの手戻り・既存機能との衝突・検証できない手順）を探し、見つけたら計画に反映してから出す。

issei の「OK」が出てから実装に入る。修正指示が入ったら計画に反映してから進む。

## 実装（自律で進める）

計画どおり実装する。既存コードのスタイル・パターンに厳密に従い、要件を忠実に（過剰実装しない）。ここは止まらず進めてよい — ただし計画から外れる判断をしたら、迷う場合は保守的な選択（変更が小さく戻しやすい方）に倒し、その判断を理由ごと Gate 4 に記録する。

## Gate 4: 知見メモ（設計〜実装を貫く継続スレッド）

これは最後の一点ではなく、**Gate 1〜5 を通してずっと回す**。issei が「設計しながら・実装しながら知見を溜める」ための中核。判断が出るたびに拾い、以下の 3 タイミングは特に取りこぼさない:

- **Gate 1** — 盲点マップで浮いた「暗黙の前提」で、issei が「知らなかった/意識してなかった」と反応したもの。
- **Gate 2** — **採用案だけでなく、捨てた案とその理由**（なぜ案 A にして案 B を捨てたか）。これは後から効く一級の知見なので必ず拾う。
- **実装中** — 当初の計画から外れた・非自明なトレードオフで決めた判断（迷って保守的な選択に倒した場合はその理由も）。

拾った知見は 2 つに仕分ける:

- **横断的に効く知見**（このプロジェクト以降も効く設計判断・ハマりどころ・「なぜこの方式か」「なぜあの方式を避けたか」）→ **memory システムに残す**。既存の memory 作法に従う（フロントマター付き 1 ファイル 1 事実・`MEMORY.md` に 1 行索引・`type` は `project` か `feedback`・関連は `[[name]]` でリンク）。**重複を避け**、既存ファイルに該当があれば新規作成せず更新する。
- **このタスク限りの細かい判断** → memory に入れず、Gate 5 の理解確認・完了報告に列挙する。

**何でも memory に書かない。** 「後から別の場面で思い出せると効くか」で選別する。コードや git 履歴を見れば分かることは書かない（memory 衛生の原則）。

## Gate 5: 理解確認（最後）

実装しっぱなしにせず、**issei が理解した状態で終わらせる**。既定は上から目線のクイズではなく腹落ちの確認（issei が望めば後述のクイズ形式に切り替える）:

1. **変更の急所を 3 点**に絞って解説する（なぜこの設計か / 触ると壊れる箇所 / 残したトレードオフ）。
2. Gate 4 で残した判断メモへのリンク（memory に書いたなら該当ファイル名）を示す。
3. 「**この 3 点、腹落ちしてる? 引っかかる所ある?**」と問い、詰まった点があれば補足する。

> issei が「クイズで確認したい」と言ったら、腹落ち確認の代わりに**設計判断の理由を問うクイズを 3 問**出す。出典は Gate 2 で比較した案と Gate 4 の知見メモに限定し（「なぜ A 方式にしたか」「ここを変えると何が壊れるか」系）、実装の字面を問う暗記クイズにしない。答えられなかった問いは解説してから終える。

ここで PR にするなら [[ship]] に渡す（banso 自体は PR を作らない）。

## やらないこと

- **PR 作成・merge はしない。** 必要なら ship に委譲する。
- **止まらず完走しない。** 5 ゲートで必ず止まる。「介入なしで一気に」を求められたらこの skill ではなく implement-issue を使う。
- **盲点マップと理解確認を省かない。** この 2 つを飛ばすと banso である意味が無い。
