# debug-issue

バグ・例外・想定外の挙動を、**いきなり修正コードに走らず**「再現 → 仮説 → 検証 → 失敗テスト先行 → 最小修正 → 適用前レビュー」の順で調査して直す。調査フェーズと修正フェーズを言葉で区切り、各段で止まってユーザーの確認を取る。対症療法でエラーメッセージだけ消すのではなく、根本原因に到達してから最小限の差分で直すのが狙い。

## 進め方（各段が終わったら止まってユーザー確認を待つ）

### Step 1: 再現 — 条件を言語化する
- バグが起きる条件を **入力・状態・手順** として書き出す（必要な前提状態: DB / 環境変数 / ログイン状態など）。
- **不明な前提は推測で埋めず質問する。** 決めつけて進まない。
- ユーザーに「手で再現できるレベル」の手順を渡し、「ここまでは再現する / しない」のフィードバックをもらって絞り込む。
- ログは **全文 + 文脈**（何のログか・いつ取れたか）で受け取る。スタックトレースは先頭から末尾まで省略しない（途中を省くと推測でフレームを補完してしまう）。長くてコンテキストを圧迫するなら、ファイルに保存してもらって `Read` で自分で読む。

### Step 2: 仮説 — 原因候補を複数、根拠つきで
- 原因の候補を **複数** 挙げ、それぞれ **コードの該当箇所（`file:line`）** を `Grep` / `Read` で特定して根拠にする。
- 「症状（エラーメッセージ）」ではなく「なぜそうなるか」を、コードを辿って説明する。

### Step 3: 検証 — 仮説を1つに切り分ける
- 各仮説を切り分けるために **どこを確認すれば良いか** を述べる（ログ追加・該当値の確認・上流の入力経路の確認など）。
- 対症療法に逃げない。例: 「`user?.id` にすればエラーは消えるが、そもそも `null` が来るのが正しいのか」まで踏み込み、**上流の根本原因**（例: 空ボディを返す関数）に到達する。

### Step 4: 失敗テストを先に書く（red を確認）
- 原因を1つに絞れたら、**バグを再現する失敗テストを1つ書く**。今は失敗する（red）ことを実行して確認し、テスト名・内容を見せて **止まる**。
- **テスト自体を書き換えて「通った」ことにしない**（Verifier Theater 対策）。テストは仕様の固定であって、いじる対象ではない。

### Step 5: 最小修正をパッチ案として提示する（まだ適用しない）
- 失敗テストが緑になる **影響範囲が最小の修正** を **パッチ案として示し、適用前に差分の意図** を説明する: どのファイルの何を・なぜ変えるのか / この修正で再現テストが通る理由 / 副作用が出そうな箇所。
- **今回の調査と無関係なリファクタは含めない**（差分を膨らませない）。
- ここで **止まってユーザーの承認を待つ**（承認前にワークツリーへ適用しない）。

### Step 6: 承認を得て適用・検証
- ユーザーが納得したら修正を適用し、`Bash` で **テスト全体を実行**して、失敗テストが緑になり他のテストを壊していないことを確認する。
- コミット・PR まで進めるなら [[ship]] に渡す（この skill 自身はコミットしない）。

## してはいけないこと

- **いきなり修正コードを書かない。** 調査（Step 1〜3）と修正（Step 4〜6）を必ず分け、各段で止まる。
- **対症療法でエラーメッセージだけ消さない。** 根本原因に到達してから直す。
- **テストを書き換えて通したことにしない**（Verifier Theater）。失敗テストは red → 修正 → green の順を守る。
- **無関係なリファクタで差分を膨らませない。** 今回のバグに必要な最小修正だけ。
- **再現条件を推測で決めつけない。** 不明な前提は質問する。
- スタックトレースを途中省略で渡さない（自分が受け取る側でも、省略されていたら全文を求める）。

## 関連

- 既存 PR を直すのは [[fix-pr]]（PR URL を受け取り worktree で修正）。debug-issue は **手元のバグ調査〜最小修正**まで。
- 新機能の実装は [[implement-issue]]、採用 / 方式の GO・NO-GO 調査は [[spike]]、実装済みコードの第三者レビューは doc-impl-reviewer agent。
- 直したものを PR にするなら [[ship]] へ。
