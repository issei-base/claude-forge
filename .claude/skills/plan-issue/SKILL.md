# Issue Planner

GitHub IssueのURLを受け取り、インタビュー→コードベース調査→計画作成→レビューループを通じて実装計画をIssueコメントとして投稿する。

**重要: 計画の最終形は GitHub Issue のコメントとして投稿する。`docs/` 等のリポジトリ管理下に MD ファイルとして残さない。** 投稿手段としての一時ファイル（`/tmp/plan_<issue>.md` 等）は構わない。

計画本文の雛形は [`references/plan-template.md`](references/plan-template.md) を参照（Issue の規模に応じてセクションを増減する）。

## ワークフロー概要

1. **Issue取得・分析** - GitHub Issue の内容を取得して把握
2. **自律調査（先行）** - 命名規則・既存パターン・他環境変数・リファレンス実装の有無を grep で確定（人間に聞く前に）
3. **インタビュー** - 自律調査で埋まらなかった「人間の意思決定が要る未確定」だけをユーザに確認（提案付き）
4. **コードベース調査（深掘り）** - 関連コード・影響範囲・呼び出し元・一次情報を調査
5. **計画コメントのドラフト作成** - 第三者が実装できる粒度でドラフトを準備（まだ投稿しない）
6. **レビュー→修正ループ** - doc-review スキルでドラフトをレビューし、PASSするまで修正
7. **Issueコメントとして投稿** - 確定したドラフトを `gh issue comment` で投稿
8. **完了報告** - 投稿したコメントURLをユーザに提示

> **調査を先に、インタビューは後に。** 命名規則・既存の設計パターン・他環境(dev/stg/prd)の変数名や値・リファレンス実装の有無のように **grep で自律的に確定できることは、人間に聞かず先に調べて埋める**。人間に聞くのは「自分で調べても決まらないこと」＝トレードオフ・優先度・対象範囲などの **意思決定**だけに絞る。
>
> **レビューは投稿前に行う。** 投稿後に PATCH で更新するのではなく、Issue 通知ノイズとレビュー履歴の散らかりを避けるため、確定版だけを1回投稿する方針にしている。

## Step 1: Issue取得・分析

引数で渡されたGitHub Issue URLから内容を取得する。

```
gh issue view <URL> --json title,body,labels,assignees,milestone,comments
```

取得したら以下を整理する:
- 要求されている機能・修正の概要
- 受け入れ条件（あれば）
- 関連する技術的キーワード
- 既存コメントの内容（対応方針が書かれていれば優先採用）

## Step 2: 自律調査（先行・人間に聞く前に）

**インタビューの前に、grep / Glob / Read で自律的に確定できることを先に潰す。** 命名規則・既存の設計パターン・他環境(dev/stg/prd)の変数名や値・リファレンス実装の有無は、人間に聞かなくてもコードを読めば分かる。聞く前に調べる。

Step 4-1 のリポジトリ準備（ローカルに clone / 最新化）を先に済ませてから、軽い調査を実施する。**ここは「確定できることを埋める」軽い調査**で、呼び出し元の網羅確認・一次情報の WebFetch などの深掘りは Step 4 で行う。

**先に確定させる観点（grep で埋まるもの）:**
- **命名規則**: 既存リソース名・変数名を grep し、実際に使われているプレフィックス・命名パターンを確定する（例: `grep -rn "resource \"" *.tf` / 既存変数名の列挙）
- **既存の設計パターン**: 同種の実装が既にあるか、どのモジュール/ディレクトリ構成・ネスト方針で書かれているか
- **他環境(dev/stg/prd)の変数名・値**: 環境別ディレクトリ / tfvars / 設定ファイルを横断して既存の変数名・値を確定する
- **リファレンス実装の有無**: 「〇〇に合わせる」対象になりうる既存実装が repo 内（または参照先 repo）に存在するか

**埋まった観点はインタビューに出さない。** 調査結果は次の Step 3 のチェックリスト消し込みに渡し、確定できた観点は質問から外す。**揃え方が一意に決まる場合は聞かない。複数あり得て人間の選好が要る場合だけ Step 3 で確認する。**

## Step 3: インタビュー

計画を練るために必要な前提情報のうち、**Step 2 の自律調査でも埋まらなかったもの**をユーザに質問する。

**質問を組み立てる前のチェックリスト消し込み（必須）:**
Issue のドメイン（Terraform / IAM / GitHub Actions / ALB / 環境変数 / アプリ実装等）を特定し、下記「ドメイン別の追加観点」から該当項目を 1 つずつ消し込んでから AskUserQuestion を組み立てる。各観点について以下を順に判断し、漏れを残さない:

1. **該当しないから外す**（ドメインに無関係）
2. **Issue に既に答えがあるから外す**
3. **調べれば分かるから外す** — **この観点は Step 2 の自律調査（grep / Read）で確定できるか？ できるなら聞かずに調査で埋める。** 命名規則・既存パターン・他環境変数・リファレンス実装の有無はここに該当することが多い。確定できたら質問から外す
4. **ユーザに聞くまでもなく明らかだから外す**
5. 上記いずれにも当たらず、**人間の意思決定（トレードオフ・優先度・対象範囲の選好）が残る**ものだけを質問に残す

> **Why:** 観点リストが列挙されていても AskUserQuestion 生成時に使い切れず投稿後に指摘されるケースが頻発するため、消し込み手順を明文化する。

**必ず確認すべき観点（共通）:**
- 技術選定に関する制約や希望（ライブラリ、フレームワーク等）
- 既存の設計方針・アーキテクチャとの整合性
- 非機能要件（パフォーマンス、セキュリティ等）
- スコープの確認（何をやらないか）
- テスト方針
- リファレンス実装の有無（存在有無は Step 2 で確定済み。**揃え方に選好が要る時だけ**確認する）
- 新規リソースの汎用化方針（特定用途専用か、汎用的に設計するか）
- 費用見積もりの要否
- **リソース・変数の命名規則** — 既存命名は Step 2 で確定済み。**揃え方が複数あり得る**（既存に揃える / 新規プレフィックス等で選好が割れる）時だけ確認する

**ドメイン別の追加観点（該当する場合に必ず確認）:**

- **Terraform / IaC**:
  - apply 挙動（in-place update か destroy/create か、`moved` ブロックの要否）
  - module 構成（既存と同じネスト/フラットに揃えるか）
  - 変数の必須/optional（デフォルト値を持たせて汎用化するか）
- **IAM ロール / セキュリティリソース**:
  - 信頼ポリシーの接続元制限（VPCe ID / `aws:SourceIp` / `aws:VpcSourceIp` 等を既存リソースから踏襲するか）
  - MaxSessionDuration / External ID 等の各フィールドを必須/optional のどちらにするか
- **GitHub Actions**:
  - 認証方式（PAT / GitHub App / OIDC のどれか、既存 App を流用できるか、Secret 命名）
  - 実装方式（`actions/github-script` / Custom JS Action / Composite Action のどれか）
  - 一次情報 URL を計画に含めるか
- **ALB / ルーティング**:
  - 既存ルールへの追記か新規ルールか（host 条件の将来削除予定も含めて確認）
  - path pattern とクエリパラメータ・末尾スラッシュの挙動
- **環境変数 / 設定値**:
  - 他環境（dev/stg/prd）の既存変数名・値は Step 2 で確定済み（質問に出さない）
  - リポジトリ間で命名が違う場合の踏襲方針（揃え方が割れる時だけ確認）
- **アプリ実装（個人開発でよく使う観点）**:
  - 既存のディレクトリ構成・モジュール分割に合わせるか
  - 外部 API / SDK のバージョン・レート制限・コストの考慮

> **Why:** これらの観点漏れにより計画投稿後に追加質問→更新ループが多発する。Issue のドメインに該当する観点はインタビュー段階で必ず潰す。

**質問のスタイル:**
- **教示先行型をデフォルトに**: 各選択肢の差分・トレードオフ・推奨理由をテキストで先に提示してから、AskUserQuestion を投げる。背景がない状態でいきなり4問セットを投げない
- 一度に大量に聞かず、優先度の高いものから2〜4個ずつ（1ターン目は最重要1問だけでもよい）
- 各選択肢に「(Recommended)」を明示し、推奨と理由を1行で添える
- ユーザの回答を受けて追加質問が必要なら続ける

**収束基準（いつ止めるか）:**
- **停止条件: 人間の意思決定が必要な未確定（トレードオフ・優先度・対象範囲など）が残っていないこと。** これらが出揃ったら、それ以上ダラダラ追加質問せず Step 4 に進む
- 「念のため」の確認を質問に積まない（調査で埋まるものは Step 2 / Step 4 へ）
- **上限の目安: 主要トレードオフが出揃えば打ち切り。** 残った細部は計画ドラフトに前提として書き、レビューで詰める

> **Why:** 質問のみ並べると、ユーザは「なぜそれを聞いているのか」が分からず clarify か interrupt で止まる。判断材料を先に渡してから選んでもらう。また、停止条件を持たないと「確認のための確認」が積み上がり、調査で済むことまで人間の時間を奪う。

## Step 4: コードベース調査（深掘り）

計画に必要なコードベースの情報をローカルリポジトリで調査する。**Step 2 で済ませた軽い自律調査（命名・既存パターン・他環境変数・リファレンス有無）の上に、呼び出し元の網羅確認・一次情報の確認などの深掘りを重ねる。**（リポジトリ準備は Step 2 で済んでいれば再実行不要。）

### 4-1: リポジトリの準備

Issue URL から owner/repo を特定し、[`_shared/pr-conventions.md`](../_shared/pr-conventions.md) **§6** の手順（カレント確認 → `~/projects/<リポジトリ名>` 探索 → 無ければ `gh repo clone` → default branch を最新化）でローカルを準備する。plan-issue は調査のみでブランチは切らない。

### 4-2: コードベース調査の実施

リポジトリのパスを作業ディレクトリとして、Agent tool（`subagent_type: Explore`）を活用して調査する。

**調査項目:**
- 変更対象となるファイル・クラス・関数の特定
- 既存の類似実装パターンの把握
- 影響を受ける依存コード（呼び出し元、テスト等）
- 設定ファイルやDB変更の必要性
- **対象ファイル/リソースの「呼び出し元・参照元」確認**（reusable workflow なら `gh search code 'uses: <owner>/<repo>/.github/workflows/<file>@'`、Terraform module なら他リポからの参照、ALB rule なら依存サービス等）。「内部だけで使われている」と独断分類しない
- **一次情報の確認**（変更対象が外部 action / AWS リソース / 公式 SDK 等の場合）。挙動が計画判断に影響する仕様（path pattern とクエリ文字列、terraform apply の進行順、action input の deprecation 状況、SHA 固定 vs タグ参照の差分など）は **WebFetch で公式ドキュメント or README を1つ以上確認し、計画コメントに URL を引用する**

> **Why:** 「呼び出し元未確認」「一次情報未確認」が原因で投稿後の全面書き直しが発生しやすい。投稿前に必ず潰す。

### 4-3: リファレンス実装との照合

Step 2 / Step 3 でリファレンス実装が特定された場合、以下を必ず確認する。

**リソース照合:**
- リファレンスに存在するリソースと今回追加するリソースを比較する
- リファレンスに**存在しない**リソースを含める場合は、その理由を計画に明記する
- リファレンスに存在するが計画に含めないリソースがあれば、除外理由を明記する

**揃える/揃えない境界の明示:**
- リファレンスと「全部揃える」ことは稀。**どの観点を揃え、どの観点は揃えないか** を計画コメントの TL;DR または 変更内容セクションに表で明示する
- 「〇〇と同じ薄い設計」のようなあいまい表現は使わない（accuracy レビューで指摘される）

**配置・命名の照合:**
- リファレンスでリソースがどのモジュール/ディレクトリに配置されているかを確認し、同じ配置パターンに合わせる
- リソース名・変数名はリファレンスの命名規則に揃える（特定用途名ではなく汎用名が使われていればそれに合わせる）

### 4-4: 既存リソースの重複・不要チェック

新規リソース追加に伴い、既存の重複・不要リソースがないかを確認する。
- 同等の機能を持つ既存リソースが別の場所にあれば、統合または削除をスコープに含める
- 未使用のリソースが見つかった場合はクリーンアップ候補として計画に含める

調査結果は計画コメントに反映する。

## Step 5: 計画コメントのドラフト作成

**ドラフトは `/tmp/plan_<issue_number>.md` に Write して、レビュー PASS 後に `gh issue comment --body-file` で投稿する。** リポジトリ管理下（`docs/` 等）にファイルを残さなければよい。

雛形は [`references/plan-template.md`](references/plan-template.md) をベースにし、Issue の規模に応じてセクションを増減する。小さなバグ修正なら「実装方針」と「テスト計画」だけで十分なこともある。

> **Why:** ドラフトを `/tmp` に置くと、wc -l でのセルフチェック・doc-review への引き渡し・--edit-last での更新が同じファイルで完結する。

### 5-1: コメント内容の作成

**実装に必要な情報だけに絞って簡潔に書く。150-200行程度を目安とする。**

**含めるセクション:**
- **TL;DR**: 対象・手段・着手順・前提などの要点を 3-5 行で
- **変更内容**: 具体的な変更 (差分・設定値・diff)。コードスニペットやコマンドはコピペで使える粒度
- **作業手順**: 順序付きの実施ステップ (環境ごとの違いは明示)
- **スコープ外**: 別 Issue で扱う事項
- **関連 Issue**: 親・前提・後続 Issue へのリンク

**原則含めないセクション (実装計画コメントは「何をどう変えるか」が主役):** [`_shared/pr-conventions.md`](../_shared/pr-conventions.md) §1「原則含めないもの」に従う（ロールバック手順/判定基準・検証チェックリスト/監視ポイント・進捗チェックリスト・詳細なリスク表・運用詳細＝ロールバック RTO / 5xx 閾値 / 監視ダッシュボード URL 等）。

> 上記の「含めない」項目は、実作業時のプレイブックや別ドキュメントで扱う。doc-review が網羅性レビューで「追加した方が良い」と返してきても、計画コメントには取り込まない。
>
> 例外: 「ピーク期を避ける」「24-48h 監視後に prd へ」のような **作業着手の前提条件** に直結する運用情報は §作業手順 内に最低限の要点だけ記載する。

**コメント作成時の原則:**
- **第三者がこのコメントだけを読んで実装できる粒度**で書く
- 「〜を参照」ではなく、参照先の内容を直接記載する（読者がローカルリポジトリにアクセスできない前提）
- コード例を含める場合は省略せず、コピペで使える粒度で記載する
- 変数名・リソース名は実際の命名規則に沿った具体名を使用する
- ファイルパスを引用する際は行番号も併記する (例: `setup-env L67-105`)

### 5-2: 構造チェック・推敲（ドラフト確定前のセルフチェック）

ドラフトを書き終えたら、Step 6（レビュー）に渡す前に以下を自己点検する。**行数指標だけでは検知できない構造的な冗長化を直前で抑えるため。**

**(A) 推敲（簡潔さ）:** [`_shared/pr-conventions.md`](../_shared/pr-conventions.md) §2「推敲」のチェックリストを通す（最低限の情報量に絞る／重複・蛇足を削る／一文で済む説明を箇条書きで膨らませない）。

**(B) 主要設計判断の確定:**
- [ ] 該当する以下の項目がすべて確定しているか:
  - 認証方式（PAT / GitHub App / OIDC）
  - 実装言語・方式（github-script / Custom JS Action / Composite Action 等）
  - リソース・変数の命名規則（既存と揃える / 新規プレフィックス）
  - セキュリティ制約の踏襲（VPCe / IP 制限 / External ID / MaxSessionDuration 等）
  - apply 挙動（in-place / destroy-create / `moved` 要否）
  - 他環境（dev/stg/prd）での既存変数名・値との整合性
- [ ] 該当するのに未確定の項目があれば、まず Step 4 の調査で埋められないか確認し、調べても決まらない（人間の選好が要る）ものだけ Step 3 に戻ってユーザに追加質問する

**(C) 構造:**
- [ ] 「原則含めないセクション」が混入していないか
- [ ] 行数が 150-200 を大幅に超えていないか（超えていたら何を削るか検討）
- [ ] §作業手順 内に「作業着手の前提条件」に直結しない運用情報が紛れ込んでいないか
- [ ] 「〜を参照」で済ませて参照先内容を本文に転記し忘れていないか

混入や未確定項目が見つかったらこの時点で対処する。混入したまま Step 6 に渡すと、レビュワー（doc-review）が ops 観点で「逆に追加した方が良い」と返してきてしまい、削るより足す方向にズレる原因になる。

## Step 6: レビュー→修正ループ（投稿前）

**Skill ツールで `doc-review` スキルを呼び出し**、**ドラフト**をレビューする。**この時点ではまだ Issue コメントに投稿していない。**

doc-review はレビュワー（観点別サブエージェント）を並列起動して観点別レビューを行う。Agent ツールで `subagent_type=general-purpose` を直接単発起動しない（doc-review の並列・観点別役割分担が機能しなくなるため）。

> **新規投稿時は doc-review を必ず実行する。既存コメント更新（`--edit-last`）の場合は、設計判断（認証方式・命名規則・セキュリティ制約・apply 挙動・他環境整合性等）に踏み込む変更を含むときのみ doc-review を実行し、typo 修正や言い回し調整等の軽微な変更ならスキップしてよい。** 判断に迷ったら実行する側に倒す。

**レビュー対象の渡し方:**
ドラフトのファイルパス（`/tmp/plan_<issue>.md`）または内容そのものをレビュー対象として渡す。「これからIssueコメントとして投稿する予定の計画」であることを明示する。

**ループの流れ:**
1. doc-review にドラフトをレビューさせる
2. 指摘事項を確認し、ドラフトを修正（`/tmp` のファイルを Edit するだけで、GitHub への通信は発生しない）
3. 再度レビューを依頼
4. レビュワーが問題なしと判断したら Step 7 へ

**レビュー指摘の取り込み方針:**
- 技術的正確性 (accuracy) / 実用性 (practicality) / 明確性 (clarity) の指摘は積極的に取り込む
- 運用観点 (ops) の指摘で「ロールバック手順を追加」「監視チェックリストを追加」など **コメントを長大化させる提案は原則取り込まない** (§5-1 の「原則含めないセクション」参照)。これらは実作業時のプレイブックや別ドキュメントで扱う旨を doc-review への返信で明示する
- ターゲット行数 150-200 行を超えたら、何を削るか検討する

レビュー中にユーザへの確認が必要な事項が出た場合は、ユーザに質問してから修正する。

### 6-2: 主要設計判断の read-back 確認（投稿直前）

doc-review PASS 後、Step 7 の `gh issue comment` を実行する直前に、計画の **主要設計判断 3-5 項目を箇条書きで再提示**し、ユーザに「この前提で投稿する」を1回確認する。

read-back 項目例（該当するもののみ）:
- スコープ（対象環境・対象リポ・対象機能）
- 認証/実装方式
- リファレンス実装との揃え方（揃える/揃えない箇所）
- 既存リソースへの追記 or 新規追加
- apply 挙動 / module 構造

ユーザが OK → Step 7 へ。ズレあり → `/tmp/plan_<issue>.md` を Edit して再 doc-review。

> **Why:** 投稿後にユーザが「やっぱりこっち」と方針反転して全面書き直しが発生しやすい。投稿直前に主要判断だけ1ターン確認すれば、書き直しコストが大幅に下がる。

## Step 7: Issueコメントとして投稿

レビュー PASS 後、確定したドラフトを 1 回だけ投稿する。

```bash
gh issue comment <Issue URL> --body-file /tmp/plan_<issue_number>.md
```

投稿後、出力からコメントURLを控えて Step 8 で提示する。

> 投稿後にユーザから指摘があった場合はコメントを更新する。更新は `gh issue comment <Issue URL> --edit-last --body-file <draft path>` を使う。
>
> **Why:** `gh api ... -X PATCH` は permission denied になりやすい。`--edit-last` なら直前に自分で投稿したコメントを確実に更新できる。
> **How to apply:** plan-issue は直前に自分で投稿したコメントを更新対象にするため `--edit-last` で問題ない。別ユーザ/別コメントを編集する必要がある特殊ケースでのみ `gh api -X PATCH ... --input <file>` を検討する。

## Step 8: 完了報告

投稿したIssueコメントのURLをユーザに提示し、計画の要約を報告する。

```
実装計画をIssueコメントに投稿しました:
- Issue: <Issue URL>
- コメント: <Comment URL>
- 概要: <計画の要約（1-2行）>
```

**派生タスクが出てきた場合の橋渡し:**

完了後、ユーザから「ついでに〇〇も対応したい」「別リポの △△ も修正が必要」と派生タスクが出てきた場合は:

1. 派生タスクのスコープを1-2行で要約し、ユーザに「これは別 Issue か、本 Issue のスコープ拡張か」を確認
2. **別 Issue 扱いなら、まず `gh issue list` / `gh issue view` で既存 Issue を検索**（重複作成・後出し回避のため）
3. 既存 Issue があれば create-issue（リライトモード）へ、なければ create-issue（新規モード）→ plan-issue へ橋渡し

## Step 9: 追加修正フェーズ（ユーザから追加質問・指摘があった場合）

> **更新コマンドは `gh issue comment <URL> --edit-last --body-file <path>` の一択。`gh api ... -X PATCH` は試さない**（permission denied で失敗しやすい。直前に自分で投稿したコメントを編集する用途では `--edit-last` で十分）。

投稿後にユーザから観点漏れ・方針変更が出てきた場合は、以下のループに入る。**「投稿で終わり」ではなく「投稿 = レビュー開始」として扱う**：

1. `/tmp/plan_<issue>.md` を Edit で更新
2. **推敲パスを必ず1回入れる** (§5-2 のセルフチェック準拠)。追記だけで終わらせず、冗長になった箇所を削る
3. 大幅な方針変更（認証方式・実装方式・命名規則の変更等）を含む場合は doc-review に再レビュー依頼
4. `gh issue comment <Issue URL> --edit-last --body-file /tmp/plan_<issue>.md` で更新
5. ユーザに更新完了を簡潔に報告（コメント URL は変わらない）

> **How to apply:** 単発の typo 修正レベルなら doc-review はスキップしてよい。設計判断に踏み込む変更（例: 認証方式の変更）の時だけ再レビュー。
