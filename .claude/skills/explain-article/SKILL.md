# explain-article

技術記事 (Zenn / Qiita / 個人ブログ / 英語記事) の URL・ローカルパス・貼り付けテキストを渡されたら、**その内容を理解できるよう、毎回ブレない深掘り型でチャットに解説する** skill。成果物は「解説そのもの」。既定の読み手は **issei 本人 (前提知識あり・自分用に深く)**。

## 何がうれしいか (素の Claude との差)

要約だけなら素の Claude で足りる。この skill の存在価値は次の 4 点を**毎回そろえる**こと:

1. **毎回ブレない深掘り型** — TL;DR → 背景 → 論理の流れ → 用語 → コード読み解き → 批判的視点 → takeaway を固定の型で出す。
2. **要約モデルを挟まない確実な本文取得** — `extract.py` / `curl` で生テキストを読む ([[doc-illustrate]] / [[cc-article]] と同じ原則)。
3. **用語・概念の解説** — 記事内の専門用語・略語・前提概念を平易に補う。
4. **鵜呑みにしない** — 主張を批判的に見て、**時間で変わる箇所だけを 2〜3 点に絞って**一次ソースで裏取りする (対象・手段はワークフロー 4)。

**この skill は理解のための解説に徹する。** 取り込み判断・実装はしない (それは [[cc-article]] / 各実装 skill)。

## 前提 (本文の取得方針)

- 記事取得は **要約モデルを挟まない生テキスト最優先**。**claude-forge 原本の絶対パス**で [[doc-illustrate]] 同梱の `extract.py` を叩く (cwd 非依存・global symlink でも効く):
  `python3 ~/projects/claude-forge/.claude/skills/doc-illustrate/scripts/extract.py <URL>`
  これは Zenn / Qiita / 一般ブログの本文をそのまま吐く (検証済み)。
- **長い記事は一度ファイルに保存してから読む** (再取得しない): `… extract.py <URL> > /tmp/article.txt` に落とし、以降は `Read` で部分読みする。本文を読み返すたびに extract.py を叩き直さない (毎回ネット往復する無駄を避ける)。
- 取れなければ `curl -sL -A "Mozilla/5.0" <URL>` で生 HTML を取り自分で読む。**WebFetch は curl も失敗した時だけの最後のフォールバック** (小型モデルがコマンド名・フラグ・バージョンを幻覚するので常用しない)。
- ローカルパスを渡されたら `Read` で読む。テキストを貼り付けられたらそのまま使う。
- **Qiita は本文の前にナビ/UI 文言** (`searchsearchSearchLogin…`、Share ボタン) が混じる。`##` 見出し以降が本文なので頭のノイズは無視する。**Zenn も**先頭に著者名・絵文字・タグ、末尾に `## Discussion` + プロフィール/免責が付く — 本文は最初の `##` 見出し〜`## Discussion` の直前まで。
- JS でしか描画されず本文が取れない / 有料・会員限定で途中で切れる場合は、ユーザーに本文の貼り付けを頼む。**本文が取れていない記事を推測で解説しない。**

## 読み手レベル (既定 = 自分用・深掘り)

- **既定は「自分用・深掘り」**: 前提知識ありを想定し、結論だけでなく「なぜそうなるか」のメカニズムまで踏み込む。批判的視点も入れる。
- ユーザーが **「受講生向けに」「初心者向けに」「やさしく」** と指定したときだけ易しめに切り替える: 用語解説を厚くし、批判的視点は浅く、たとえ話で補う。図解 HTML 教材が欲しいなら [[doc-illustrate]] に渡す。

## ワークフロー

### 1. 本文を生テキストで取得する
上記「前提」の手段で本文を取る。取れなければ貼り付けを頼む。

### 2. 記事の幹を掴む
何の記事か / 主張・手法の核 / 論理の流れ (なぜその結論に至るか) / 前提 (バージョン・OS・依存) / 専門用語 / コード・具体例。

### 3. 深掘り型で解説を組み立てる
下記「出力フォーマット」の順で書く。**結論だけ並べず、各要点は「なぜそうなるか」まで噛み砕く。** ただし**深さは記事の長さ・複雑さに合わせる**: 短い / 単純な記事は節を畳み、コードが無ければ「コード・具体例の読み解き」節は出さない。**空の節を記事外の一般論で埋めない (水増し禁止)。**

### 4. 批判的に見て、変わりうる主張だけ裏取りする
まず記事の **前提の限界・反証・再現性**を一度疑う。そのうえで裏取りするのは次の**時間で変わる主張のみ・多くても 2〜3 点に絞る** (それ以外は裏取りしない):
- バージョン依存の挙動 / コマンド・フラグ / 数値・上限・価格 / 「新機能」主張 / 既に過ぎた日付

裏取りは一次ソースで行う。**`WebSearch` は公式ページ・changelog の URL を探すだけに使い、検索 snippet を根拠にしない** (snippet も要約モデルの産物)。見つけた公式 URL は本文取得と同じ生テキスト経路 (`extract.py` / `curl`) で読み、その本文を根拠に「現行で有効 / 古い可能性」と一言添える。**Claude / Claude Code の主張は `docs.claude.com` / `code.claude.com` を直接当てる**。AWS の挙動・上限・価格は [[aws-docs]] に寄せる。`claude-api` skill / `claude-code-guide` agent が使える環境なら、そちらの方が速い。

## 出力フォーマット (チャット・日本語)

```
## 📄 <記事タイトル>
<出典: ホスト名 (URL から)。著者・公開日は本文に明記がある時だけ併記し、無ければ書かない (推測で埋めない — extract.py はメタ情報を落とすので、見当たらなければ省く)>

**ひとことで**: <1〜2 文。記事の主張の核>

### 背景・前提
<なぜこの記事が書かれたか / 前提となる文脈・問題意識を一段で>

### 論理の流れ（要点）
<記事の主張・手法を、結論だけでなく「なぜそうなるか」まで噛み砕いて順に 3〜6 点>

### 用語・概念
<記事内の専門用語・略語・前提概念を 1 行ずつ平易に。理解の鍵になるものに絞る>

### コード・具体例の読み解き（あれば）
<記事のコード / コマンド / 図を、何をしているか・なぜそう書くかで噛み砕く>

### 批判的に見ると
<鵜呑みにしない: 前提の限界 / 別の見方・反証 / 再現性 / 「古い可能性」(必要なら一次ソースで裏取りした結果)>

### 自分はどう使うか（takeaway）
<自分の文脈での持ち帰り 1〜3 点。CC 環境への取り込み判断が要るなら [[cc-article]] に渡すと一言>
```

受講生向けに切り替えたときは「批判的に見ると」を浅くし、「用語・概念」を厚くする。

## してはいけないこと

- **本文を取らずに推測で解説しない。** `extract.py` / `curl` / `Read` で本文を取る。取れなければ貼り付けを頼む。
- **要約モデルの産物を根拠にしない。** 本文は生テキストで読む (WebFetch の要約で済ませない)。裏取りでも **WebSearch の snippet を根拠にせず**、見つけた公式 URL を生テキストで読み直す。
- **裏取りを全部 or ゼロにしない。** 時間で変わる主張だけを 2〜3 点に絞って一次ソースで確認する (対象カテゴリと手段はワークフロー 4)。古い / 不確かなら明示する。
- **節を水増ししない。** 記事に該当が無い節は出さない・空の節を記事外の一般論で埋めない (深さは記事に合わせる)。
- **取り込み診断・実装・図解 HTML 化はしない。** 理解のための解説に徹する。取り込み判断は [[cc-article]]、図解 HTML 教材は [[doc-illustrate]]、音声・動画・PDF・画像は [[gemini-multimodal]] に渡す。
- 記事や URL に**受講生の個人情報・非公開 URL**が含まれていたら、解説にもチャットにも残さない。

## 関連

- 取得・正確さの原則は [[doc-illustrate]] / [[cc-article]] / [[aws-docs]] と同じ (生テキスト優先・主張は一次ソースで裏取り)。`extract.py` も [[doc-illustrate]] のものを流用。
- **役割の住み分け**: explain-article = 記事の**中身を理解する**解説、[[cc-article]] = 記事を**自分の CC 環境に取り込めるか診断**、[[doc-illustrate]] = 公式 docs を**図解 HTML 教材化**、[[gemini-multimodal]] = 音声・動画・PDF・画像の解析。
- skill を新設したら **README の「現在の skill」表に 1 行追記** (claude-forge の約束事)。
