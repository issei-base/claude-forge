# ビジュアルスタイル指針

`assets/template.html` の各部品を「いつ・どう使うか」の解説。狙いは **受講生がスクロールするだけで要点をつかめる、手描き風の温かい1枚もの**。教科書のような文字びっしりではなく、図解と余白でリズムを作る。

## 全体トーン

- 温かいクリーム背景・丸ゴシック・マーカー強調の「手書きノート」感。冷たい・お役所的な見た目にしない。
- **1コンセプト = 1ブロック**。難しい概念ほど図解・比喩・吹き出しで噛み砕く。
- 文章は短く。1段落は2〜4文。専門用語は初出で必ず噛み砕く（受講生向け）。
- 励ます語り口を1〜2か所入れる（例:「まずはここから！」「触れば一気に分かる」）。サンプルの owl/rabbit 注釈の役割。
- 絵文字はアイコン代わりに効果的に。ただし1ブロックに詰め込みすぎない。

## 部品の使い分け

| 部品 | クラス | いつ使う |
|---|---|---|
| 表紙 | `.hero` | 必ず先頭に1つ。h1 は「一番言いたいこと」を比喩で。`.marker` で1語強調 |
| 結論ボックス | `.tldr` | 必ず hero 直後に1つ。**結論を最初に**。「で、何が嬉しい/何ができる」を2〜3文 |
| 特徴カード | `.features`+`.feat` | 「ポイントは3つ」を①②③で。2〜4枚。多すぎたらセクションに分ける |
| セクション見出し | `.section`+`.sec-num` | 本文の章立て。番号は 01,02… の連番。`.en` に英語正式名 |
| 吹き出し | `.bubble` | たとえ話・つまずき・コツの一言。各セクションに0〜1個まで（多用すると散らかる） |
| ステップ | `.steps` | 手順・流れ・順番のあるもの |
| 比較2カラム | `.compare` | A vs B / 向き不向き / before-after |
| 逐語引用 | `.quote` | **上限・数値・コマンド構文・パラメータ名**。1〜数行のベタ書きで足りるとき。言い換えずそのまま（後述） |
| データ表 | `.dtable` | 上限・料金・比較が「項目 × 値」の**格子**になるとき（料金倍率、ストレージクラス別、クォータ一覧 等）。数値セルは `class="num"` で強調。値は原典のまま逐語 |
| コールアウト | `.callout note`/`.callout tip` | note=注意/落とし穴(ピンク)、tip=コツ/受講生への一言(緑) |
| ゲージ | `.gauge` | 連続スペクトル（安い⇔速い、軽い⇔深い 等）を見せたい時 **だけ**。針角度は `rotate(角度 200 170)` を -75〜75 で調整 |
| 出典 | `.source` | **必ず末尾に**。一次ソースへのリンク + 生成日 |

## 構成の型（おすすめの並び）

```
hero → tldr → features(任意) → section×3〜6（各 section に本文＋必要なら bubble/steps/compare/quote/callout）→ source
```

全体で **スクロール 1〜3 画面**に収まる密度がちょうどいい。長文ページでも「受講生が最初に知るべき幹」に絞る。枝葉は原典リンクに任せる。

## 図解（SVG/CSS）を起こすときの発想

サンプルの「天秤」「ダイヤル」のように、**概念を物理メタファに置き換える**と一気に分かりやすくなる。コードで描けるものを優先:

- **天秤/シーソー** … トレードオフ（価格⇔価値、速度⇔精度）。SVG の `<line>`+`<rect>` で支点と皿。
- **ゲージ/ダイヤル** … 連続スペクトル。`.gauge` をそのまま流用。
- **箱/容器** … 容量・格納（S3、キュー）。角丸 `<rect>` + 中身の点々。
- **矢印フロー** … 処理の流れ（リクエスト→Lambda→DB）。`.steps` か横並びカード+「→」。
- **積み重ね/階層** … レイヤー構造（VPC→Subnet→EC2）。入れ子の `<rect>`。
- **円グラフ/割合** … `stroke-dasharray` の円。

凝った絵を描き込もうとしない。**単純な幾何 + ラベル + 1個の比喩**で十分伝わる。描くのが難しければ無理に SVG にせず、`.features` カードや `.compare` で代替する。

## 事実の正確さ（最重要）

図解は理解を助けるためで、**内容を盛る・推測で埋めるためではない**。

- **数値・上限・クォータ・コマンド構文・パラメータ名は、原典の表現をそのまま** `.quote` に入れる。言い換えると AWS/Anthropic が値を変えたとき嘘になる。[[aws-docs]] と同じ原則。
- 原典に書いていないことは書かない。「たぶんこうだろう」を断定で書かない。比喩はあくまで比喩とわかる形で。
- 出典リンクは必須。`.source` の `__SOURCE_URL__` を実際の原典 URL にする。
- 専門用語を噛み砕くのはOKだが、**事実を単純化しすぎて誤りにしない**。曖昧なら原典の言い回しに寄せる。

## 配色・フォント（テンプレ済み・基本いじらない）

- 配色は CSS 変数（`--accent` 等）で一元管理済み。分野で色を変えたい場合のみ `--accent` を差し替える（例: Claude/Anthropic 系は `--accent:#C15F3C` の珊瑚色など）。やりすぎ注意。
- フォントは Google Fonts（Yusei Magic / Zen Maru Gothic / Yomogi）。オフラインでも丸ゴシック系にフォールバックするよう設定済み。受講生に配る前提でも単一HTMLで完結する。

## やりがちな失敗

- 文字を詰め込みすぎて「ただの白黒文書」になる → 図解・カード・余白でリズムを。
- 吹き出し・絵文字の多用で散らかる → 各セクション最小限。
- ゲージを無理に使う → 連続量がないなら使わない。
- 出典を消す/ぼかす → 必ず原典リンクを残す。
- 原典の数値を「約」「だいたい」で丸める → 逐語で。
