# doc-illustrate

AWS / Anthropic の公式技術ページを、**手描き風の図解 HTML** に要約する skill。文字びっしりの要約ではなく、受講生がスクロールするだけで幹をつかめる「見て分かる 1 枚もの」を作るのが狙い。

同梱物（必ず使う。毎回 CSS や構成を作り直さない）:
- `assets/template.html` — 配色・フォント・部品（hero / 結論ボックス / ①②③カード / 吹き出し / ステップ / 比較 / ゲージ / 逐語引用 / 出典）が揃った共通デザインシステム。**コピーして `<body>` の中身だけ差し替える**（CSS は触らない）。
- `assets/svg-snippets.html` — 図解 SVG の部品カタログ（矢印 marker / 横フロー / 入れ子 / 1→多 分岐 / グループ箱 / 階層スタック）。**アーキ図はフルスクラッチせず、ここからコピーして座標とラベルだけ調整**（座標バグと時間を両方減らせる）。
- `scripts/extract.py` — URL を渡すと原典を「生テキスト」で吐く（AWS=本文抽出 / Mintlify=`.md` / その他=タグ剥がし を自動判別）。ワークフロー1の決定論版。
- `scripts/verify.py` — 生成 HTML の `.quote` / `td.num` を原典と機械照合する。ワークフロー3.5の必須ゲートの自動版。
- `references/visual-style.md` — 各部品の使い分けと、図解の起こし方・事実の正確さの指針。**HTML を組む前に必ず読む**。
- `references/example-output.html` — 完成見本（Claude Opus 4.8 発表ページを図解化したもの）。密度・トーン・セクション数（スクロール 1〜3 画面）の基準。**初めて作る時は一度開いて感覚を掴む**。

## Preflight

1. **対象 URL を取得**（引数 or プロンプト）。なければユーザーに聞く。
2. **保存先フォルダ**: 既定は `~/Downloads/`（repo 外なので公開リスクが無く、生成物をすぐ開ける・受講生に渡しやすい）。ユーザー指定があればそちら。`mkdir -p ~/Downloads` で用意（通常は既にある）。**claude-forge は public repo なので、repo 内の追跡される場所には絶対に保存しない**（既定の `~/Downloads` は repo 外なので安全。出力先を変える時も gitignore 済み or repo 外に）。
3. ソース種別を判定し、取得手段を選ぶ。**正確さ最優先 = 要約モデルを挟まない「生テキスト」が最良**（下記は検証済み）。**いずれの種別も `python3 scripts/extract.py <URL>` がそのまま生テキストを吐く**（種別を自動判別）。手で取りたい時の内訳は下記:
   - `docs.aws.amazon.com` など AWS → `aws` MCP の `read_documentation` が**使えれば**それが一次ソース。**MCP が無ければ即 `scripts/extract.py`（内部で `main-col-body` を抽出）か下の `curl`**。MCP の有無で迷って空振りしない。
   - `docs.claude.com` / `docs.anthropic.com`（Claude 開発者ドキュメント・Mintlify 製）→ **URL 末尾に `.md` を付けて `curl`** すると整形済み markdown がそのまま返る（例: `…/models/overview.md`）。全ページ索引は `https://docs.claude.com/llms.txt`（全文ダンプ `llms-full.txt` は約 71MB なので grep 用途以外で丸ごと取らない）。
   - `anthropic.com/news` などマーケティング/発表ページ → **`curl -sL -A "Mozilla/5.0"` で生 HTML** を取り、自分で読む。価格・ベンチ数値・モデル ID は生 HTML に逐語で入っており、`grep -oE` で抜き出せる。
   - **WebFetch は最後の手段**。間に小型の要約モデルが挟まり、**数値・価格・固有名・存在しない機能まで幻覚する**（実例: 実在しない製品名や倍率を生成）。WebFetch 単独で数値を確定しないこと（→ ワークフロー 1 と 3.5 の検証を必ず通す）。
   - **クロスホストのリダイレクト**: `curl -sL` は自動追従するので楽。WebFetch を使う場合のみ手動で追う（別ホストへの 301 はリンクが返るだけで本文は来ない）。

## ワークフロー

### 1. 原典を読む（生テキストで）
- **既定は `python3 scripts/extract.py <URL> > /tmp/source.txt`**。これで AWS=本文抽出 / Claude docs=`.md` / 発表ページ=タグ剥がし を自動でやり、生テキストが手に入る（この `source.txt` はワークフロー3.5の照合にそのまま使う）。MCP `read_documentation` が使える AWS は併用してよい。WebFetch を使ったら、その出力は**下書き扱い**にして数値は生ソースで裏取りする。
- **出典リンクはユーザーが踏む正規の入口（元の URL）**を使う。`.md` や `curl` で取った中身でも、`.source` に載せる URL は人間が開く通常版。
- **AWS の `ja_jp` は機械翻訳**（原典冒頭に「齟齬があれば英語版が優先」と明記）。引用文が不自然・曖昧なら、URL の `ja_jp` を `en_us` に替えた英語原典で技術用語を裏取りする。図解の本文は日本語でよいが、**パラメータ名・API 名・enum 値などの厳密な綴りは英語原典が正**。
- ページの**幹**を掴む: 何のサービス/機能か、何が嬉しいか、要点 3 つ、つまずきやすい点、**重要な数値・上限・コマンド構文・パラメータ名**。
- **肝心の数値が画像/グラフの中にある時**（Anthropic 発表ページのベンチ比較表など、テキストに数字が無い）→ ① 本文プロースに出ている数値だけ逐語採用 ② 表の数字は載せない ③「詳細スコアは原典の図を参照」と注意書きに明記 ④ どうしても要るならユーザーにスクショ提供を依頼。**画像内の数字を推測で書かない。**
- **重要な“図そのもの”が画像のとき**（AWS のアーキテクチャ図など）→ **本文プロースに根拠がある構造だけ** `assets/svg-snippets.html` の部品で自分で描き起こしてよい（例: 「リスナー→ルール→ターゲットグループ」と本文に書いてあれば図にできる）。**本文に根拠の無い細部（具体的なポート番号・台数など）は描かず**、「詳細は原典の図を参照」とリンクで委ねる。
- 受講生が最初に知るべき核に絞る。枝葉は原典リンクに委ねる（全部詰め込まない）。

### 2. スタイル指針を読む
- `references/visual-style.md` を読み、構成の型（hero → 結論 → 特徴 → セクション → 出典）と部品の使い分け、図解メタファ、**事実の正確さルール**を頭に入れる。

### 3. HTML を組む
- **まず丸ごとコピー**: `cp assets/template.html <保存先>/<slug>-<生成日>.html`。**`<style>` は一切手で打ち直さない**（CSS をインラインで再入力するとデザインシステムがドリフトする）。以降は Edit で `<body>` とプレースホルダだけを差し替える。
- プレースホルダを埋める: `__TITLE__` / `__SOURCE_URL__`（正規の入口 URL）/ `__SOURCE_TITLE__` / `__SOURCE_PUBLISHER__`（発行元。例「AWS 公式ドキュメント」「Claude・Anthropic 公式ドキュメント」）/ `__PUBLISH_DATE__`（原典の公開日。無ければ削除）/ `__DATE__`（この資料の生成日＝今日。環境の現在日付を使う、`YYYY-MM-DD`）。**公開日と生成日は別物**なので混ぜない。
- 使わない部品は削除、必要なら同じクラスで増やす。料金・上限・クォータなど「項目×値」の格子は `.dtable`（表）で。難しい概念は**比喩図（天秤・ダイヤル・箱・矢印フロー等）**で噛み砕く。**アーキ図・フロー図は `assets/svg-snippets.html` の部品をコピーして座標とラベルだけ調整**（フルスクラッチは座標バグの元）。コードで描けない凝った絵は無理に作らずカード/比較で代替。
- **図解を必ず入れる（この skill の肝）**: テキストの箇条書きで済ませない。原典が説明している **分類・階層・関係・処理の流れ・トレードオフ** を、最低 1 つ（理想は 2〜3 つ）図に起こして記事を**補足**する — 全体像ツリー / 横フロー / 入れ子 / 1→多 分岐 / 比較 2 カラム / 天秤・ゲージ等のメタファ。原典が「文字だけ」でも、本文に書かれた関係はこちらで図に翻訳してよい（むしろそれが価値）。`assets/svg-snippets.html` の部品をコピーして座標・ラベルだけ調整する。**ただし図にしてよいのは「原典のテキストに根拠がある構造」だけ**。本文に無い細部（具体的な数値・台数・ポート等）は描かず「詳細は原典を参照」に委ねる（→「してはいけないこと」の事実を盛らない原則は不変）。組んだ後、図のラベルが枠からはみ出していないかはワークフロー 4 の QA で必ず確認する。
- **配色を分野色に変えたい時は CSS 変数（`:root` の `--accent` / `--accent-deep`）だけ差し替える**（例: Claude/Anthropic 系は `--accent:#C15F3C` の珊瑚色）。ゲージ等の SVG は `var(--accent)` 参照済みなので追従する（インラインに hex を直書きしない）。
- **数値・上限・syntax は `.quote` に逐語**で。言い換えない（[[aws-docs]] と同じ原則）。原典に無いことは書かない。

### 3.5. 数値を検証する（保存前の必須ゲート）
- 組んだ HTML の **`.quote` と `.dtable` の数値・価格・上限・モデル ID・コマンド構文を、原典の生テキストと 1 つずつ突き合わせる**。WebFetch を使った場合は特に必須（要約モデルの幻覚を必ずここで潰す）。
- **まず自動照合**: `python3 scripts/verify.py <生成HTML> /tmp/source.txt`。`.quote` / `td.num` を抜き、数値・「…」で括った語・英数字識別子（=ハードトークン）が原典に在るかを空白無視で照合する。`✗ FAIL` が出たら**その数値/識別子は原典に無い** → 消すか原典の表記に直す（終了コードも 1）。`⚠ 要確認` は言い回し差の可能性なので、その箇所だけ原典と目視照合。
- スクリプトが拾えない言い換え部分は、生テキストを `grep` して該当文字列が原典に在ることを目視で確認する。**「だいたい合ってる」で通さない。**

### 4. 見た目を確認して保存・公開する
- ファイル名は内容スラッグ + 生成日（例: `s3-overview-2026-05-31.html`）。既定の `~/Downloads/` へ書き出す。同名が既にあれば末尾に `-2` 等を付けて上書きを避ける。
- **ビジュアル QA（“見て分かる”が目的なので省かない）**: 保存したら一度レンダリングを目視する。特に**手で描いた SVG はテキストはみ出し・要素のはみ出し・重なりが起きやすい**。webapp-testing skill（Playwright・外部プラグイン。導入済みのときだけ）でスクショを撮って崩れを確認するのが確実。少なくとも一度はブラウザで開いて、図のラベルが枠から出ていないか・全体が縦 1〜3 画面に収まっているかを見る。崩れていたら座標やラベル長を直す。
- **保存できたら自動でブラウザで開く（必須ステップ）**: macOS は `open <path>`、Linux は `xdg-open <path>` を実行する。ブラウザを開く手段が無い環境（ヘッドレス等）の時だけ、開けなかった旨と保存先パスを案内する。
- ユーザーに **保存先パス（`~/Downloads/...`）と要約の構成（どのセクションを立て、どんな図解を起こしたか）** と、**画像内の数値を割愛した等の正確さ上の判断**を報告し、「気になる所を直す/別の見せ方にする?」と一声かける。

## してはいけないこと

- **数値・上限・クォータ・コマンド構文・パラメータ名を言い換えない。** 逐語で `.quote` に。AWS/Anthropic はこれらを変える。**保存前に必ずワークフロー 3.5 の検証を通す。**
- **WebFetch の要約をそのまま事実として書かない。** 小型モデルが数値・価格・固有名を幻覚する。生テキスト（`curl` / `.md` / MCP）で裏取りしてから載せる。
- **画像/グラフの中の数字を推測で書かない。** テキストに無いスコアは載せず、「詳細は原典の図を参照」と明記する。
- **原典に無い情報を図解で“盛らない”。** 比喩で噛み砕くのは可だが、推測を断定で書かない。曖昧なら原典の言い回しに寄せる。
- **出典リンクを必ず残す。** `.source` を消さない・ぼかさない。出典なしの技術主張は NG。
- **生成 HTML を git に commit しない。** 既定の出力先 `~/Downloads/` は repo 外（public repo に追跡させると公開されてしまうため）。出力先を claude-forge 内に変える場合は必ず `.gitignore` 済みの場所にし、追跡される場所には絶対に置かない。
- 受講生の個人情報や非公開 URL を永続 memory / repo に保存しない。
- 単一 HTML で完結させる。重い外部依存を足さない（フォントの Google Fonts CDN のみ可、フォールバック有り）。
