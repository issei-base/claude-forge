# tdd（テストを先に落とす）

issue フローの外でコードを書くときの規律。**テストが仕様の写し**であり、先に落ちるテストが「何を作るか」を確定させる。実装はそれを通すためだけに書く。

## 手順

1. **振る舞いを先に列挙する。** 実装に入る前に、期待する振る舞いを 1 行 1 ケースの自然言語で列挙してユーザに見せる（正常系 / 境界 / エラー系。3〜10 個目安）。このリストが仕様になる — ユーザは割り込みで差し替え・追加できる。**列挙を飛ばしてコードを書き始めない**
2. **repo のテスト流儀を確認する。** 既存テストの置き場所・ランナー・命名・モック方針を読み、同じ流儀で書く（新しいテスト基盤を勝手に持ち込まない）。**「テストが無い」という依頼の前提は鵜呑みにしない** — 対象のシンボル名で repo 全体を grep し、同ディレクトリ以外（契約テスト・別パッケージのミラー等）に既存カバレッジが無いか確認する。見つかったら新規追加ではなく既存テストの拡張、または依頼者への事実確認を優先する
3. **テストを書いて落とす（red）。** 手順 1 のケースをテストコードに落とし、実装前に実行して**落ちることを確認する**。落ちないテストは振る舞いを検証できていないので書き直す。**バグ修正では「バグを再現するテスト」が red** — 再現テストなしにバグを直さない。**対象がまだ存在しない新規実装では**、先にコンパイルは通るが正しい値を返さないダミー実装を用意してから実行する — import/モジュール未解決エラーによる一律の赤は red に数えず、各ケースが期待値との不一致という意味のある理由で個別に落ちて初めて red とする
4. **最小の実装で通す（green）。** テストを通すことだけを目的に実装する。テストが要求しない機能を足さない
5. **通ったまま整える（refactor）。** green を保ったまま重複・命名・構造を直す。このあいだテストは触らない
6. **報告に red→green を含める。** 「先に落とした件数 → 通した件数」を完了報告に書く。スキップ・例外があれば理由ごと申告する

## 例外（この 3 つだけ・すべて申告必須）

- **回帰固定テスト**: 既に正しく動いている振る舞いをそのまま固定するテスト（契約テスト・ゴールデン・スナップショット初回）は落とせない。初回 green で正当 — ただし「対象の振る舞いが変更前から実装済みだった」根拠（変更前の該当ファイル・行）を報告に添える。根拠を示せない申告は回帰固定と認めず、通常の red→green の対象とする。**発見のトリガー**: 手順 3 で実装を何も変えていないのに green になったら、それが回帰固定テストの合図 — その場で根拠を確認し、示せなければ実装側の見落としを疑って red に戻す
- **テスト基盤が無い repo**: 手順 3〜5 をスキップし、報告に明記する
- **テストで検証できない変更**: docs・設定値（コード分岐を伴わない）・コメントのみ。UI の見た目でもスナップショット / ビジュアル回帰で検証できるならスキップしない。**「設定値」の判定は字面でなく実体で行う** — 変更対象の値の使用箇所を grep し、比較演算子や if 文の条件式に使われていれば「コード分岐を伴う」ので対象外（通常の red→green）。しきい値・タイムアウト・上限件数など「数値を変えるだけ」に見える依頼ほど先に確認する

## やらないこと

- **テストの後付けをしない。** 実装が済んでから「動作確認としてテストを書く」のは TDD ではない — その順になりそうなら手順 1 に戻る
- **green を偽装しない。** 通らないテストの skip 化・アサートの緩和・期待値の書き換えで green にしない。テストが間違っていた場合は、その判断を明示してから直す。**テストは正しかったが仕様が意図的に変わった**場合（しきい値変更など）は「テストが間違っていた」と偽らず、新しい仕様での red を一度示してから期待値を更新し、変更理由（仕様変更）を明示する
- **数行の自明な修正でも省略しない。** 最低限「どの既存テストがこの変更を担保するか」を 1 行で言う。担保するテストが無ければそれが手順 3 の対象
