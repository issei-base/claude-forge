# tdd（テストを先に落とす）

issue フローの外でコードを書くときの規律。**テストが仕様の写し**であり、先に落ちるテストが「何を作るか」を確定させる。実装はそれを通すためだけに書く。

## 手順

1. **振る舞いを先に列挙する。** 実装に入る前に、期待する振る舞いを 1 行 1 ケースの自然言語で列挙してユーザに見せる（正常系 / 境界 / エラー系。3〜10 個目安）。このリストが仕様になる — ユーザは割り込みで差し替え・追加できる。**列挙を飛ばしてコードを書き始めない**
2. **repo のテスト流儀を確認する。** 既存テストの置き場所・ランナー・命名・モック方針を読み、同じ流儀で書く（新しいテスト基盤を勝手に持ち込まない）
3. **テストを書いて落とす（red）。** 手順 1 のケースをテストコードに落とし、実装前に実行して**落ちることを確認する**。落ちないテストは振る舞いを検証できていないので書き直す。**バグ修正では「バグを再現するテスト」が red** — 再現テストなしにバグを直さない
4. **最小の実装で通す（green）。** テストを通すことだけを目的に実装する。テストが要求しない機能を足さない
5. **通ったまま整える（refactor）。** green を保ったまま重複・命名・構造を直す。このあいだテストは触らない
6. **報告に red→green を含める。** 「先に落とした件数 → 通した件数」を完了報告に書く。スキップ・例外があれば理由ごと申告する

## 例外（この 3 つだけ・すべて申告必須）

- **回帰固定テスト**: 既に正しく動いている振る舞いをそのまま固定するテスト（契約テスト・ゴールデン・スナップショット初回）は落とせない。初回 green で正当 — ただし「対象の振る舞いが変更前から実装済みだった」根拠（変更前の該当ファイル・行）を報告に添える。根拠を示せない申告は回帰固定と認めず、通常の red→green の対象とする
- **テスト基盤が無い repo**: 手順 3〜5 をスキップし、報告に明記する
- **テストで検証できない変更**: docs・設定値（コード分岐を伴わない）・コメントのみ。UI の見た目でもスナップショット / ビジュアル回帰で検証できるならスキップしない

## やらないこと

- **テストの後付けをしない。** 実装が済んでから「動作確認としてテストを書く」のは TDD ではない — その順になりそうなら手順 1 に戻る
- **green を偽装しない。** 通らないテストの skip 化・アサートの緩和・期待値の書き換えで green にしない。テストが間違っていた場合は、その判断を明示してから直す
- **数行の自明な修正でも省略しない。** 最低限「どの既存テストがこの変更を担保するか」を 1 行で言う。担保するテストが無ければそれが手順 3 の対象
