# spike

技術的な問い・選択に対して、**実装に入る前に「進む / やめる」を決めるための判断材料**を出す。timebox を切って調べ、`GO` / `NO-GO` / `INCONCLUSIVE` の判定を根拠付きで返す。成果物は **意思決定ドキュメントであって実装計画ではない**。

由来は「spike」(XP 由来の、不確実性を潰すための短い使い捨て調査)。**何を作るかが決まった後の実装計画は [[plan-issue]] の仕事**。spike はその一段上流 ——「そもそも作るべきか / どの方式を採るか」を決める。

## いつ使う / 使わない

- **使う**: 方式の分岐 (A か B か)、実現可能性の不安 (この制約下で成立するか)、ライブラリ / API が要件を満たすかの検証、採用是非の意思決定。
- **使わない**:
  - 作ると既に決まっていて実装手順が欲しい → [[plan-issue]] / [[implement-issue]]。
  - AWS の仕様・上限・syntax をただ引きたい → [[aws-docs]]。
  - 調査の結果として判断 (GO/NO-GO) が要らない、ただの情報収集 → 素の Claude で足りる。

## 原則: timebox を切る

spike の価値は **「決めるのに十分な証拠」を最短で集めること**であって、網羅的な調査ではない。だから最初にスコープと打ち切り条件を決める。証拠が決定に十分なら、まだ調べられることが残っていても止めて判定を出す。逆に決定的な証拠が出ないなら無理に GO/NO-GO をでっち上げず `INCONCLUSIVE` とし、**何を追加で潰せば決まるか**を書く (これも立派な成果)。

## 手順

### 1. 問いを固める

- ユーザーの問いを **検証可能な命題**に言い換える (例:「リアルタイム通知に WebSocket と SSE どっちが良い?」→「我々の制約 [想定同時接続 N・既存スタック・運用体制] 下で、双方向性が必須か / SSE で足りるか」)。
- **制約・成功条件・打ち切り条件**を明示する。問いが曖昧なら、ここで一度ユーザーに確認する (前提を勝手に埋めない)。

### 2. 証拠を集める (timebox 内)

ソースを適切に選ぶ:

- **コードベース** — 既存実装・依存・制約は `Read` / `Grep` / `Glob` で一次確認する。
- **ドキュメント / Web** — `WebSearch` / `WebFetch` で公式仕様・上限・既知の落とし穴を引く。CC / ライブラリのバージョン依存に注意 (主張は一次ソースで裏取り)。
- **AWS が絡む論点** — 記憶で書かず [[aws-docs]] に寄せる (一次ソース原則)。
- **小さく試す (任意)** — 安く確かめられるなら使い捨ての検証コードを書いて走らせる。**スクラッチ領域 (`/tmp` 等) に置き、対象 repo にはコミットしない**。これが spike の "spike" たる所以。

### 3. 敵対的チェック (設計分岐・実現性リスクがあるとき推奨)

有力案に乗る前に「これが NO-GO になるとしたら何が原因か」を能動的に潰す。

- 別モデルの独立した目が欲しければ、有力案の差分や設計メモを [[codex-secondopinion]] に渡して adversarial に突いてもらう。
- それ自体が大げさなら、自分で「最も痛い失敗モード」を列挙して各々に証拠で当たる。

### 4. 判定を出す

下記テンプレートで **チャットに**出す (既定)。ユーザーが「ファイルにして」と言ったときだけ `~/Downloads` か指定パスに書き出す (claude-forge は生成物を repo に置かない方針)。

```
# Spike: <問い>

## 判定: GO / NO-GO / INCONCLUSIVE
<1〜2 文の推奨と、その理由の核>

## 検討した選択肢
- <案 A>: <要点>
- <案 B>: <要点>

## 根拠 (証拠と出典)
- <事実 / 計測 / 仕様>（出典: <URL / file:line>）
- ...

## トレードオフ
<案ごとの効きどころと痛みどころ>

## 残リスク・未確認事項
<判定の前提・まだ潰せていない不確実性>

## 次の一歩
<GO なら: 実装計画は plan-issue / 着手は implement-issue。NO-GO なら: 代替や撤退。INCONCLUSIVE なら: 何を追加調査すれば決まるか>
```

## してはいけないこと

- **網羅調査に発散しない。** timebox を超えたら、足りなくても現時点の証拠で判定 (または `INCONCLUSIVE`) を出す。
- **証拠の無い GO/NO-GO を出さない。** 推奨は必ず根拠 (出典付き) に紐付ける。決め手が無いなら `INCONCLUSIVE`。
- **実装計画にすり替えない。** タスク分解や具体実装が欲しくなったら、それは [[plan-issue]] / [[implement-issue]] の領分。spike は判断材料まで。
- **検証コードを対象 repo に残さない。** 使い捨ては使い捨てとして scratch に置く。

## 関連

- 作ると決めた後の実装計画 → [[plan-issue]]、着手 → [[implement-issue]]。
- 有力案の敵対的レビュー → [[codex-secondopinion]]。
- AWS の一次ソース確認 → [[aws-docs]]。
